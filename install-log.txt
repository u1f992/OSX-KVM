Sequoiaのインストールは可能なものの、無操作時にもCPU 100%に張り付く挙動が見られる。
以下の情報によると、`halt_poll_ns`が影響している可能性がある。

- https://forum.proxmox.com/threads/kvm-high-cpu-usage-even-without-starting-any-vm.93066/
- https://github.com/GNS3/gns3-vm/issues/128

$ cat /sys/module/kvm/parameters/halt_poll_ns
200000
$ echo 0 | sudo tee /sys/module/kvm/parameters/halt_poll_ns

halt_poll_ns=0は効果なし。

APICvかと推測したが、i9-9900KF + Z390チップセットはPosted Interruptsを
ハードウェアサポートしていない（コンシューマ向けCPUの制限）。
vmx flagsにposted_intr, vapic_reg, vidがない。
$ grep 'vmx flags' /proc/cpuinfo | head -1 | tr ' ' '\n' | grep -iE 'posted_intr|vapic_reg|vid'
(empty)
$ cat /sys/module/kvm_intel/parameters/enable_apicv
N
enable_apicv=1を指定しても、ハードウェアが非対応のためNのまま。

perf kvm stat liveでVM exit内訳を確認:

Sequoiaデスクトップ (idle):
  MSR_WRITE ~53% (~8000/sec), IO_INSTRUCTION ~33% (~5000/sec), HLT ~12%
  Total: ~15000 exits/sec

Recovery環境 (idle):
  MSR_WRITE ~62% (~430/sec), HLT ~17%, EPT_MISCONFIG ~15%
  IO_INSTRUCTION: 0
  Total: ~700 exits/sec

Recovery環境ではVM exitが約20分の1で、vCPU使用率は0に漸近する。
フルmacOS起動後にMSR_WRITEとIO_INSTRUCTIONが大量発生するのは
macOSのデーモンやドライバ（電力管理、WindowServer等）によるもの。
APICvなしではこれらのVM exitは回避不可能なハードウェア制約。
ゲスト内Activity Monitorではidle 90%以上であり、実負荷には影響しない。

Windows VMではHyper-V enlightenments（hv-vapic, hv-stimer等）がAPIC操作や
タイマーを準仮想化し、APICvなしでもVM exitを大幅に削減する。
macOSにはこの仕組みがないため、同一ハードウェアでもホスト側CPU消費が大きく異なる。

TahoeはUpdate Mac Automaticallyでハングしてしまうが、
これは明らかに最後の画面であり、インストール自体は完了していると仮定してよいだろう。
十分まってディスクIOがないことを確認した上で、根本的な解決ではない前提でresetをかけてみる

virsh --connect qemu:///system reset macOS

実際にログイン画面まで遷移できる。
ログイン後はKeyboard Setup Assistantが表示される。これはSequoiaのインストール正常完了後と同じ状況

Tahoeの古いインストールならどうか？26.1は取得できた

fetch-macOS-v2で取得できるものは最新のBaseSystem

- Tahoeの最新のBaseSystemでTahoe26.1のpkgをオフラインインストール→失敗「An Internet connection is required to install macOS.」
- SequoiaからTahoe26.1のpkgでインストール→同上

SequoiaのBaseSystemにもバックポート更新が入っていると仮定すると、
BaseSystemより古いインストーラは使えない（インターネット接続＝更新を強制する）のではと考えられる
