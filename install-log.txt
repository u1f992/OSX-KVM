Sequoiaのインストールは可能なものの、無操作時にもCPU 100%に張り付く挙動が見られる。
以下の情報によると、`halt_poll_ns`が影響している可能性がある。

- https://forum.proxmox.com/threads/kvm-high-cpu-usage-even-without-starting-any-vm.93066/
- https://github.com/GNS3/gns3-vm/issues/128

$ cat /sys/module/kvm/parameters/halt_poll_ns
200000
$ echo 0 | sudo tee /sys/module/kvm/parameters/halt_poll_ns

halt_poll_ns=0は効果なし。

APICvかと推測したが、i9-9900KF + Z390チップセットはPosted Interruptsを
ハードウェアサポートしていない（コンシューマ向けCPUの制限）。
vmx flagsにposted_intr, vapic_reg, vidがない。
$ grep 'vmx flags' /proc/cpuinfo | head -1 | tr ' ' '\n' | grep -iE 'posted_intr|vapic_reg|vid'
(empty)
$ cat /sys/module/kvm_intel/parameters/enable_apicv
N
enable_apicv=1を指定しても、ハードウェアが非対応のためNのまま。

perf kvm stat liveでVM exit内訳を確認:

Sequoiaデスクトップ (idle):
  MSR_WRITE ~53% (~8000/sec), IO_INSTRUCTION ~33% (~5000/sec), HLT ~12%
  Total: ~15000 exits/sec

Recovery環境 (idle):
  MSR_WRITE ~62% (~430/sec), HLT ~17%, EPT_MISCONFIG ~15%
  IO_INSTRUCTION: 0
  Total: ~700 exits/sec

Recovery環境ではVM exitが約20分の1で、vCPU使用率は0に漸近する。
フルmacOS起動後にMSR_WRITEとIO_INSTRUCTIONが大量発生するのは
macOSのデーモンやドライバ（電力管理、WindowServer等）によるもの。
APICvなしではこれらのVM exitは回避不可能なハードウェア制約。
ゲスト内Activity Monitorではidle 90%以上であり、実負荷には影響しない。

Windows VMではHyper-V enlightenments（hv-vapic, hv-stimer等）がAPIC操作や
タイマーを準仮想化し、APICvなしでもVM exitを大幅に削減する。
macOSにはこの仕組みがないため、同一ハードウェアでもホスト側CPU消費が大きく異なる。

そもそもCODEとVARSが整合していないのではという仮説を立てた。フォークした時点のupstream（[4c378a4](https://github.com/kholia/OSX-KVM/tree/4c378a4b5e0b219783683012bec680325eb40719)）で[OVMF-notes.txt](https://github.com/kholia/OSX-KVM/blob/4c378a4b5e0b219783683012bec680325eb40719/OVMF-notes.txt#L3)が参照しているovmf-generic_2025.11-3_all.debに対応する（はず）のovmf_2025.11-3_all.debには、次の文言がある。

```
$ curl -sL https://snapshot.debian.org/archive/debian/20260109T022551Z/pool/main/e/edk2/ovmf_2025.11-3_all.deb \
    | bsdtar -xOf - data.tar.xz \
    | bsdtar -xOf - ./usr/share/doc/ovmf/NEWS.Debian.gz \
    | zcat \
    | grep -A1 "2MB VAR"
  2MB VAR images are not compatible with 4MB CODE images. Users must
  migrate both CODE and VARS images simultaneously.
```

VARSはバリアントのようにいくつか同梱されているが、これらは実は全部同じで、ovmf_2023.05-2_all.debまで含まれていたものと一致する2M用のものであることがわかった。これはCODEが4Mになっているのと不一致。

```
$ curl -sL https://snapshot.debian.org/archive/debian/20260109T022551Z/pool/main/e/edk2/ovmf-generic_2025.11-3_all.deb \
  | bsdtar -xOf - data.tar.xz \

```

つまり、対応する4M用のOVMF_VARSをupstreamから取得して使えば整合し、自動選択の保存が可能になるはずだ。

```
$ curl -sL https://snapshot.debian.org/archive/debian/20260109T022551Z/pool/main/e/edk2/ovmf-generic_2025.11-3_all.deb \
  | bsdtar -xOf - data.tar.xz \
  | xz -dc \
  | bsdtar -xOf - ./usr/share/OVMF/OVMF_VARS_4M.fd \
  > / ... /OSX-KVM/OVMF_VARS_4M.fd
$ sudo setfacl -m u:libvirt-qemu:rw / ... /OSX-KVM/OVMF_VARS_4M.fd
$ virsh --connect qemu:///system edit macOS

$ stat / ... /OSX-KVM/OVMF_VARS_4M.fd  # Modify時刻をメモ
$ virsh --connect qemu:///system start macOS  # Ctrl+EnterでMacintosh HDを選択
$ stat / ... /OSX-KVM/OVMF_VARS_4M.fd  # Modifyを比較
```

デフォルトの設定は当然fdへの書き込みを伴う

```
$ git update-index --skip-worktree OVMF_VARS_4M.fd
```
